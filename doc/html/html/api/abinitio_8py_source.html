270  """ Read the reference ab initio data from a file such as qdata.txt.
271 
272  @todo Add an option for picking any slice out of
273  qdata.txt, helpful for cross-validation
274 
275  @todo Closer integration of reference data with program -
276  leave behind the qdata.txt format? (For now, I like the
277  readability of qdata.txt)
278 
279  After reading in the information from qdata.txt, it is converted
280  into the GROMACS energy units (kind of an arbitrary choice);
281  forces (kind of a misnomer in qdata.txt) are multipled by -1
282  to convert gradients to forces.
283 
284  We also subtract out the mean energies of all energy arrays
285  because energy/force matching does not account for zero-point
286  energy differences between MM and QM (i.e. energy of electrons
287  in core orbitals).
288 
289  The configurations in force/energy matching typically come
290  from a the thermodynamic ensemble of the MM force field at
291  some temperature (by running MD, for example), and for many
292  reasons it is helpful to introduce non-Boltzmann weights in
293  front of these configurations. There are two options: WHAM
294  Boltzmann weights (for combining the weights of several
295  simulations together) and QM Boltzmann weights (for converting
296  MM weights into QM weights). Note that the two sets of weights
297  'stack'; i.e. they can be used at the same time.
298 
299  A 'hybrid' ensemble is possible where we use 50% MM and 50% QM
300  weights. Please read more in LPW and Troy Van Voorhis, JCP
301  Vol. 133, Pg. 231101 (2010), doi:10.1063/1.3519043.
302 
303  @todo The WHAM Boltzmann weights are generated by external
304  scripts (wanalyze.py and make-wham-data.sh) and passed in;
305  perhaps these scripts can be added to the ForceBalance
306  distribution or integrated more tightly.
307 
308  Finally, note that using non-Boltzmann weights degrades the
309  statistical information content of the snapshots. This
310  problem will generally become worse if the ensemble to which
311  we're reweighting is dramatically different from the one we're
312  sampling from. We end up with a set of Boltzmann weights like
313  [1e-9, 1e-9, 1.0, 1e-9, 1e-9 ... ] and this is essentially just
314  one snapshot. I believe Troy is working on something to cure
315  this problem.
316 
317  Here, we have a measure for the information content of our snapshots,
318  which comes easily from the definition of information entropy:
319 
320  S = -1*Sum_i(P_i*log(P_i))
321  InfoContent = exp(-S)
322 
323  With uniform weights, InfoContent is equal to the number of snapshots;
324  with horrible weights, InfoContent is closer to one.
325

522  def get_energy_force(self, mvals, AGrad=False, AHess=False):
523  """
524  LPW 06-30-2013
525 
526  This subroutine builds the objective function (and optionally
527  its derivatives) from a general simulation software. This is
528  in contrast to using GROMACS-X2, which computes the objective
529  function and prints it out; then 'get' only needs to call
530  GROMACS and read it in.
531 
532  This subroutine interfaces with simulation software 'drivers'.
533  The driver is only expected to give the energy and forces.
534 
535  Now this subroutine may sound trivial since the objective
536  function is simply a least-squares quantity (M-Q)^2 - but
537  there are a number of nontrivial considerations. I will list
538  them here.
539 
540  0) Polytensor formulation: Because there may exist covariance
541  between different components of the force (or covariance
542  between the energy and the force), we build the objective
543  function by taking outer products of vectors that have the
544  form [E F_1x F_1y F_1z F_2x F_2y ... ], and then we trace it
545  with the inverse of the covariance matrix to get the objective
546  function.
547 
548  This version implements both the polytensor formulation and the standard formulation.
549 
550  1) Boltzmann weights and normalization: Each snapshot has its
551  own Boltzmann weight, which may or may not be normalized.
552  This subroutine does the normalization automatically.
553 
554  2) Subtracting out the mean energy gap: The zero-point energy
555  difference between reference data and simulation is
556  meaningless. This subroutine subtracts it out.
557 
558  3) Hybrid ensembles: This program builds a combined objective
559  function from both MM and QM ensembles, which is rigorously
560  better than using a single ensemble.
561 
562  Note that this subroutine does not do EVERYTHING that
563  GROMACS-X2 can do, which includes:
564 
565  1) Internal coordinate systems
566  2) 'Sampling correction' (deprecated, since it doesn't seem to work)
567  3) Analytic derivatives
