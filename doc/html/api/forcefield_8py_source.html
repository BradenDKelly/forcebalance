00001 """ @package forcebalance.forcefield
00002 
00003 Force field module.
00004 
00005 In ForceBalance a 'force field' is built from a set of files containing
00006 physical parameters.  These files can be anything that enter into any
00007 computation - our original program was quite dependent on the GROMACS
00008 force field format, but this program is set up to allow very general
00009 input formats.
00010 
00011 We introduce several important concepts:
00012 
00013 1) Adjustable parameters are allocated into a vector.
00014 
00015 To cast the force field optimization as a math problem, we treat all
00016 of the parameters on equal footing and write them as indices in a parameter vector.
00017 
00018 2) A mapping from interaction type to parameter number.
00019 
00020 Each element in the parameter vector corresponds to one or more
00021 interaction types.  Whenever we change the parameter vector and
00022 recompute the objective function, this amounts to changing the
00023 physical parameters in the simulations, so we print out new
00024 force field files for external programs.  In addition, when
00025 these programs are computing the objective function we are often
00026 in low-level subroutines that compute terms in the energy and
00027 force.  If we need an analytic derivative of the objective
00028 function, then these subroutines need to know which index of the
00029 parameter vector needs to be modified.
00030 
00031 This is done by way of a hash table: for example, when we are
00032 computing a Coulomb interaction between atom 4 and atom 5, we
00033 can build the words 'COUL4' and 'COUL5' and look it up in the
00034 parameter map; this gives us two numbers (say, 10 and 11)
00035 corresponding to the eleventh and twelfth element of the
00036 parameter vector.  Then we can compute the derivatives of
00037 the energy w/r.t. these parameters (in this case, COUL5/rij
00038 and COUL4/rij) and increment these values in the objective function
00039 gradient.
00040 
00041 In custom-implemented force fields (see counterpoisematch.py)
00042 the hash table can also be used to look up parameter values for
00043 computation of interactions.  This is probably not the fastest way
00044 to do things, however.
00045 
00046 3) Distinction between physical and mathematical parameters.
00047 
00048 The optimization algorithm works in a space that is related to, but
00049 not exactly the same as the physical parameter space.  The reasons
00050 for why we do this are:
00051 
00052 a) Each parameter has its own physical units.  On the one hand
00053 it's not right to treat different physical units all on the same
00054 footing, so nondimensionalization is desirable.  To make matters
00055 worse, the force field parameters can be small as 1e-8 or as
00056 large as 1e+6 depending on the parameter type.  This means the
00057 elements of the objective function gradient / Hessian have
00058 elements that differ from each other in size by 10+ orders of
00059 magnitude, leading to mathematical instabilities in the optimizer.
00060 
00061 b) The parameter space can be constrained, most notably for atomic
00062 partial charges where we don't want to change the overall charge
00063 on a molecule.  Thus we wish to project out certain movements
00064 in the mathematical parameters such that they don't change the physical
00065 parameters.
00066 
00067 c) We wish to regularize our optimization so as to avoid changing
00068 our parameters in very insensitive directions (linear dependencies).
00069 However, the sensitivity of the objective function to changes in the
00070 force field depends on the physical units!
00071 
00072 For all of these reasons, we introduce a 'transformation matrix'
00073 which maps mathematical parameters onto physical parameters.  The
00074 diagonal elements in this matrix are rescaling factors; they take the
00075 mathematical parameter and magnify it by this constant factor.  The
00076 off-diagonal elements correspond to rotations and other linear
00077 transformations, and currently I just use them to project out the
00078 'increase the net charge' direction in the physical parameter space.
00079 
00080 Note that with regularization, these rescaling factors are equivalent
00081 to the widths of prior distributions in a maximum likelihood framework.
00082 Because there is such a correspondence between rescaling factors and
00083 choosing a prior, they need to be chosen carefully.  This is work in
00084 progress.  Another possibility is to sample the width of the priors from
00085 a noninformative distribution -- the hyperprior (we can choose the
00086 Jeffreys prior or something).  This is work in progress.
00087 
00088 Right now only GROMACS parameters are supported, but this class is extensible,
00089 we need more modules!
00090 
00091 @author Lee-Ping Wang
00092 @date 04/2012
00093

00140     """ Determine the type of a force field file.  It is possible to
00141     specify the file type explicitly in the input file using the
00142     syntax 'force_field.ext:type'.  Otherwise this function will try
00143     to determine the force field type by extension. """
00144 
00145     fsplit = ffname.split('/')[-1].split(':')
00146     fftype = None
00147     if verbose: logger.info("Determining file type of %s ..." % fsplit[0])
00148     if len(fsplit) == 2:
00149         if fsplit[1] in FF_IOModules:
00150             if verbose: logger.info("We're golden! (%s)\n" % fsplit[1])
00151             fftype = fsplit[1]
00152         else:
00153             if verbose: logger.info("\x1b[91m Warning: \x1b[0m %s not in supported types (%s)!\n" % (fsplit[1],', '.join(FF_IOModules.keys())))
00154     elif len(fsplit) == 1:
00155         if verbose: logger.info("Guessing from extension (you may specify type with filename:type) ...")
00156         ffname = fsplit[0]
00157         ffext = ffname.split('.')[-1]
00158         if ffext in FF_Extensions:
00159             guesstype = FF_Extensions[ffext]
00160             if guesstype in FF_IOModules:
00161                 if verbose: logger.info("guessing %s -> %s!\n" % (ffext, guesstype))
00162                 fftype = guesstype
00163             else:
00164                 if verbose: logger.info("\x1b[91m Warning: \x1b[0m %s not in supported types (%s)!\n" % (fsplit[0],', '.join(FF_IOModules.keys())))

00166             if verbose: logger.info("\x1b[91m Warning: \x1b[0m %s not in supported extensions (%s)!\n" % (ffext,', '.join(FF_Extensions.keys())))

00168         if verbose: logger.warning("Force field type not determined!\n")

00320         """ Parse a force field file and add it to the class.
00321 
00322         First, figure out the type of force field file.  This is done
00323         either by explicitly specifying the type using for example,
00324         <tt> ffname force_field.xml:openmm </tt> or we figure it out
00325         by looking at the file extension.
00326 
00327         Next, parse the file.  Currently we support two classes of
00328         files - text and XML.  The two types are treated very
00329         differently; for XML we use the parsers in libxml (via the
00330         python lxml module), and for text files we have our own
00331         in-house parsing class.  Within text files, there is also a
00332         specialized GROMACS and TINKER parser as well as a generic
00333         text parser.
00334 
00335         The job of the parser is to determine the following things:
00336         1) Read the user-specified selection of parameters being fitted
00337         2) Build a mapping (dictionary) of <tt> parameter identifier -> index in parameter vector </tt>
00338         3) Build a list of physical parameter values
00339         4) Figure out where to replace the parameter values in the force field file when the values are changed
00340         5) Figure out which parameters need to be repeated or sign-flipped
00341         
00342         Generally speaking, each parameter value in the force field
00343         file has a <tt> unique parameter identifier <tt>.  The
00344         identifier consists of three parts - the interaction type, the
00345         parameter subtype (within that interaction type), and the
00346         atoms involved.
00347 
00348         --- If XML: ---
00349 
00350         The force field file is read in using the lxml Python module.  Specify
00351         which parameter you want to fit using by adding a 'parameterize' element
00352         to the end of the force field XML file, like so.
00353 
00354         @code
00355         <AmoebaVdwForce type="BUFFERED-14-7">
00356            <Vdw class="74" sigma="0.2655" epsilon="0.056484" reduction="0.910" parameterize="sigma, epsilon, reduction" /> 
00357         @endcode
00358 
00359         In this example, the parameter identifier would look like <tt> Vdw/74/epsilon </tt>.
00360 
00361         --- If GROMACS (.itp) or TINKER (.prm) : ---
00362 
00363         Follow the rules in the ITP_Reader or Tinker_Reader derived
00364         class.  Read the documentation in the class documentation or
00365         the 'feed' method to learn more.  In all cases the parameter
00366         is tagged using <tt> # PARM 3 </tt> (where # denotes a comment,
00367         the word PARM stays the same, and 3 is the field number starting
00368         from zero.)
00369 
00370         --- If normal text : ---
00371         
00372         The parameter identifier is simply built using the file name,
00373         line number, and field.  Thus, the identifier is unique but
00374         completely noninformative (which is not ideal for our
00375         purposes, but it works.)
00376 
00377         --- Endif ---

00380         If we use CHARMM and GROMACS to perform simulations as part of the same TARGET, we will
00381         get messed up.  Maybe this needs to be fixed in the future, with program prefixes to
00382         parameters like C_ , G_ .. or simply unit conversions, you get the idea.
00383 
00384         @warning I don't think the multiplier actually works for analytic derivatives unless
00385         the interaction calculator knows the multiplier as well.  I'm sure I can make this

00905     def mktransmat(self):
00906         """ Create the transformation matrix to rescale and rotate the mathematical parameters.
00907

00909         change the total charge.
00910         
00911         First build these:
00912         
00913         'qmap'    : Just a list of parameter indices that point to charges.
00914

00916                     A potential target for the molecule-specific thang.
00917                     
00918         Then make this:
00919         
00920         'qtrans2' : A transformation matrix that rotates the charge parameters.
00921                     The first row is all zeros (because it corresponds to increasing the charge on all atoms)
00922                     The other rows correspond to changing one of the parameters and decreasing all of the others
00923                     equally such that the overall charge is preserved.
00924                     
00925         'qmat2'   : An identity matrix with 'qtrans2' pasted into the right place
00926         
00927         'transmat': 'qmat2' with rows and columns scaled using self.rs
00928         
00929         'excision': Parameter indices that need to be 'cut out' because they are irrelevant and
00930                     mess with the matrix diagonalization
00931         
00932         @todo Only project out changes in total charge of a molecule, and perhaps generalize to
00933         fragments of molecules or other types of parameters.
00934         @todo The AMOEBA selection of charge depends not only on the atom type, but what that atom is bonded to.
